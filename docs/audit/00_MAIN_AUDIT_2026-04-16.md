# Repository Audit (Ist-Zustand) - 2026-04-16

## Ziel und Scope
Technisches Ist-Audit fuer den kompletten Projektordner `MealMate` als Entscheidungsgrundlage fuer eine spaetere vollstaendige Repo-Politur (Public-GitHub-tauglich).  
In diesem Dokument: Fakten, Bewertung, Entscheidung, Zielbild.  
Keine strukturellen Umbauten in diesem Audit-Block.

## Methodik (nur lesend)
1. Vollstaendige Struktur- und Dateitypanalyse inkl. Volumen/Artefakte.
2. Pruefung von Einstieg, Runtime-Konfiguration, Deployment-Dateien, Tests, Doku.
3. Git-Tracking-Pruefung fuer Public-Repo-Risiken (welche Dateien sind wirklich versioniert).
4. Konsolidierte Qualitaetsbewertung aus Sicht Engineering, Reviewer, externer Nutzer.

## Harte Fakten
1. Gesamt im Ordner: `1404` Verzeichnisse, `10285` Dateien (hauptsaechlich durch `.venv`).
2. Ohne `.venv/.git/.pytest_cache`: klarer Python-FastAPI-Stack mit `app/`, `alembic/`, `tests/`, `docs/`, `scripts/`, `tools/`.
3. `489` `__pycache__`-Verzeichnisse vorhanden (lokaler Build-Muell im Arbeitsordner).
4. Root ist stark ueberladen:
   - viele `DELIVERABLE_*.md` (teils mehrere hundert KB),
   - grosse Datendateien (`rezepte_kochwiki_clean_3713.csv` ~5.9 MB),
   - PDF-Artefakt (`projekt_3_meal_mate.pdf`),
   - lokale DB-Dateien (`mealmate.db*`, `tmp_dbg.db`).
5. Git-tracked Datei-Set: `216` Dateien.  
   Positiv: `.env`, lokale `.db`, `backups/`, `outbox/` sind nicht versioniert.  
   Negativ: diagnostische Laufartefakte in `diagnostics/` sind versioniert.
6. Konfigurations-/Branding-Inkonsistenz:
   - README nennt Produkt `MealMate` ([README.md](../../README.md):1),
   - `app_name` wird intern auf `Kitchen Hell and Heaven` normalisiert ([app/config.py](../../app/config.py):11, [app/config.py](../../app/config.py):64),
   - technische Doku fuehrt alten Repo-Pfad/Fokus ([docs/TECHNICAL_DOCUMENTATION.md](../TECHNICAL_DOCUMENTATION.md):7).
7. Lokale Startumgebung im Ordner ist inkonsistent:
   - `.venv/pyvenv.cfg` referenziert fremde/alte Pfade ([.venv/pyvenv.cfg](../../.venv/pyvenv.cfg):4).
8. Codebasis ist funktional umfangreich und nicht trivial:
   - grosse Kernmodule (`app/translation_service.py` 1162 LOC, `app/routers/recipes.py` 977 LOC, `app/services.py` 830 LOC).
9. Deploy-Bausteine sind vorhanden: `Dockerfile`, `docker-compose.yml`, `render.yaml`, `start.sh`, Alembic-Migrationen.
10. Testsuite ist gross, aber lokal im aktuellen Interpreter nicht sofort ausfuehrbar (fehlendes `pytest` im globalen Python; lokale `.venv` defekt).

## Qualitaetsbewertung (hart)
### Was professionell ist
1. Solider Tech-Stack (FastAPI, SQLAlchemy 2, Alembic, Security-Middleware, RBAC, Tests).
2. Sinnvolle Laufzeitdateien fuer Deployment vorhanden.
3. Breite funktionale Testabdeckung im Repository erkennbar.
4. Strukturierte Domainthemen im Code (`routers`, `i18n`, `security`, `translation`, `moderation`).

### Was fuer ein Public Repo unprofessionell wirkt
1. Root wirkt wie Arbeitsablage statt Produkt-Repository (Deliverables, PDF, grosse CSV, viele thematische README-Satelliten).
2. Diagnose-/Laufartefakte im Tracking (`diagnostics/*.json|*.md`) statt reproduzierbarer Reports.
3. Deutliche Naming-/Branding-Drift (`MealMate` vs `Kitchen Hell and Heaven`).
4. Historisch gewachsene Monolith-Dateien mit hoher LOC und geringer Modulgrenze.
5. Leere/unklare Strukturreste (`app/services/` Verzeichnis neben `app/services.py`).
6. Lokaler Muell im Projektordner (`.venv`, `__pycache__`, `.pytest_cache`) erschwert externe Wahrnehmung.
7. Onboarding wirkt nicht konsolidiert: viele Readmes, keine klare "single source of truth" fuer Setup + Betrieb + Entwicklung.

## Harte Problemliste
1. Root-Ueberfrachtung mit artefaktartigen Dokumenten und grossen Dateien.
2. Inkonsistente Produktidentitaet in Code und Doku.
3. Fehlende klare Trennung zwischen Runtime-Code, interner Nachweisdokumentation und Diagnose-Ausgaben.
4. Versionierte Diagnostik-Ergebnisse statt reproduzierbarer Pipeline.
5. Uneinheitliche Dokumentationslandschaft (mehrere README-Varianten ohne strikte Verantwortungsgrenzen).
6. Potenziell wartungsriskante Modulgroessen in Kernlogik.
7. Lokale Entwicklerumgebung im Ordner aktuell nicht reproduzierbar (defekte `.venv`-Referenz).
8. Public-Repo-Standardisierung unvollstaendig (klare Contribution/Support/Governance-Story nicht konsolidiert).

## Rettungsentscheidung
## Entscheidung: Variante B
Bestehendes Projekt ist **teilweise gut rettbar**, braucht aber einen **starken strukturellen Umbau mit Repo-Neurahmung**.

### Begruendung
1. Der produktive Kern ist klar vorhanden und technisch tragfaehig.
2. Die aktuelle Repo-Praesentation ist fuer externen Eindruck klar unter Niveau.
3. Ein kompletter Neuaufbau (Variante C) waere fachlich nicht noetig, weil viel Substanz vorhanden ist.
4. Ein "leichter" Umbau (Variante A) reicht nicht, da Root, Doku, Artefaktlage und Naming-Grundprobleme systemisch sind.

## Zielbild (Public-Repo)
1. Repo wirkt wie bewusst neu aufgesetzt: klarer Root, eindeutige Struktur, minimierte Historienartefakte.
2. Ein zentrales Einstiegssystem:
   - `README.md` (Produkt, Quickstart, Architektur-Short, Deployment),
   - `docs/` fuer tiefe Technikdoku,
   - `docs/internal/` fuer interne Audit-/Migrationsnotizen.
3. Strikte Trennung:
   - Runtime (`app`, `alembic`, notwendige Deploy-Dateien),
   - Dev/QA (`tests`, `scripts`, `tools`),
   - keine Laufartefakte im Tracking.
4. Bereinigte Produktidentitaet (Name, Defaults, Doku, UI, Config konsistent).
5. Reproduzierbare Start-/Testpfade ohne lokale Altumgebung.

## Empfohlener Start fuer naechsten Block

2. Danach **Naming/Config-Konsolidierung** (eine Produktidentitaet).
3. Danach **Code-Schnitt-Neuordnung** fuer uebergrosse Module.
